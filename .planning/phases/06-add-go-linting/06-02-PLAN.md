From RESEARCH.md — errcheck fix patterns:

Pattern 1: fmt.Fprintf in SSE handler (chatlogic.go)
```go
// BEFORE (triggers errcheck)
fmt.Fprintf(w, "data: %s\n\n", delta.Content)

// AFTER — write errors in SSE mean connection is closed; stop streaming
if _, err := fmt.Fprintf(w, "data: %s\n\n", delta.Content); err != nil {
    return // connection closed; stop streaming
}
```

Pattern 2: defer rows.Close() in production code (sqlitestore.go)
```go
// BEFORE (triggers errcheck)
defer rows.Close()

// AFTER — log the close error; cannot propagate from defer
defer func() {
    if err := rows.Close(); err != nil {
        log.Printf("rows.Close: %v", err)
    }
}()
```

Pattern 3: defer rows.Close() in test files where error is ignorable
```go
// AFTER — nolint with justification
defer rows.Close() //nolint:errcheck // cleanup in defer; error logged by sql driver
```

Note: For each fmt.Fprintf call in chatlogic.go, decide case-by-case:
- Token/content SSE writes: check error, return to stop streaming
- Error SSE writes (sending error to client): check error, log if write fails, then return anyway
