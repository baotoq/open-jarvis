# Phase 2: Conversation Persistence - Context

**Gathered:** 2026-03-11
**Status:** Ready for planning

<domain>
## Phase Boundary

SQLite-backed persistence for conversations (CHAT-03) and a sidebar UI to browse and resume past conversations (UI-02). Users can close the browser and reopen to find their conversation intact; a sidebar lists all past conversations for switching. Tool execution, agent capabilities, and new API features are out of scope.

</domain>

<decisions>
## Implementation Decisions

### Sidebar design
- Each conversation entry shows: title + relative date (e.g. "Fix my Go code · 2 hours ago")
- Fixed sidebar, always visible — no collapse toggle. Width: ~240-260px
- Flat list, newest first — no date grouping headers
- Hover/context menu to delete a conversation; no rename action for now

### Session identity
- Active session ID stored in `localStorage` — persists across tab closes and restarts
- On load: if stored session ID no longer exists in DB (e.g. was deleted), silently auto-create a new conversation and clear the stale ID
- New session IDs are generated by the backend on first message save (UUID); frontend receives the ID back in the response and stores it in localStorage

### Conversation title
- Auto-titled from the first user message — truncate to ~50 chars
- No AI-generated title (avoids extra latency and API call)
- No rename action in this phase

### Claude's Discretion
- Exact SQLite schema (columns, indexes, migrations)
- ConvStore replacement strategy (interface swap vs new implementation)
- Exact sidebar CSS/Tailwind styling (colors, spacing, hover states)
- Loading state while fetching conversation list on load
- Error state if SQLite connection fails at startup

</decisions>

<specifics>
## Specific Ideas

- "Title + relative date" style matches ChatGPT/Claude.ai sidebar — familiar pattern
- Backend generates session IDs (not frontend) — keeps ID authority server-side, consistent with Phase 1 design intent

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ConvStore` (`src/internal/svc/convstore.go`): in-memory map implementation. SQLite store should expose the same `Get`/`Set` interface — drop-in replacement in `ServiceContext`
- `ServiceContext` (`src/internal/svc/servicecontext.go`): injects `*ConvStore` — replace with an interface so both in-memory and SQLite stores satisfy it
- `ChatRequest` (`src/internal/types/types.go`): already carries `SessionId string` — no protocol change needed for Phase 2

### Established Patterns
- go-zero Handler → Logic → ServiceContext layer rule: logic accesses the store via ServiceContext, handler does not
- Config via go-zero YAML struct tags — SQLite file path should be a new config field (e.g. `DBPath string`)
- Tests use `testify/assert` and `require`; store swap should keep existing convstore tests green

### Integration Points
- `NewServiceContext` in `servicecontext.go`: replace `NewConvStore()` call with the SQLite-backed store when `DBPath` is configured
- Phase 2 also adds new HTTP endpoints for: list conversations, get conversation by ID, delete conversation — these follow the existing handler/logic pattern
- Frontend: sidebar is a new component alongside the existing chat area; `localStorage` read happens at app init

</code_context>
